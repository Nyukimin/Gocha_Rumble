# Gocha Rumble 3D マネタイズ・広告実装計画

**作成日**: 2026-01-13
**概要**: Google AdSenseを活用した「常時表示」の実装方針と、ゲーム性を損なわないための配置・併用案の定義。

---

## 1. 広告サービスの選定

Webブラウザゲーム（React + Three.js）の場合、選択肢は以下の通りとなる。

*   **メイン**: **Google AdSense**
    *   **ディスプレイ広告**: バナー等の常時表示用。
    *   **AdSense for Games (H5)**: ゲーム向け動画広告（インタースティシャル/リワード）。※利用にはGoogleへの申請と審査が必要な場合がある。
*   **実装**: Web標準のタグ埋め込み、またはReact用ラッパーライブラリを使用。

---

## 2. 「常時表示」の実装・配置パターン

ゲームの「没入感（1000体の群れ）」と「収益」を両立させるため、ゲーム画面（Canvas）の上には極力広告を重ねない方針を推奨する。

### 2.1 パターンA：レターボックス配置（PC推奨・最適解）
ゲーム画面のアスペクト比を固定（例: 16:9）し、ブラウザウィンドウとの差分で生まれる「余白」に広告を配置する。

*   **レイアウト構成**:
    *   **中央**: ゲーム画面 (Canvas)
    *   **左右**: 余白スペース → **ここに縦長スカイスクレイパー広告を常時表示**
    *   **下部**: 余白スペース → **横長バナー広告**
*   **メリット**: ゲームUIと一切被らないため、誤クリックも起きにくく、プレイヤーのストレスが最小限。
*   **デメリット**: スマホ（縦持ち）ではスペースが確保しづらい。

### 2.2 パターンB：アンカー広告（スマホ推奨）
Google AdSenseの「自動広告（アンカー）」機能を使用する。

*   **挙動**: 画面の上端または下端に固定され、ユーザーがスクロールしてもついてくる（閉じるボタンあり）。
*   **メリット**: 実装が一番簡単（タグを貼るだけ）。モバイルでの収益性が高い。
*   **デメリット**: HUD（UI）と被る可能性があるため、ゲーム側のUI設計で「画面下部50px〜100pxは操作エリアにしない」等の調整が必須。

### 2.3 パターンC：UI組み込み（ロビー・リザルト限定）
バトル中（観戦中）は広告を消し、ロビーやリザルト画面のUIの一部として広告枠を用意する。

*   **配置場所**:
    *   **ロビー**: 画面下部フッター付近。
    *   **リザルト**: 報酬獲得ボタンの下など。
*   **メリット**: ゲームの没入感を完全に守れる。
*   **デメリット**: バトル時間が長いと広告表示時間が減るため、収益機会が減る。

## 3. 推奨レイアウト構成案

レスポンシブ対応として、画面サイズに応じてレイアウトを切り替える構成とする。

### Reactコンポーネント構成イメージ
```tsx
<div className="app-container" style={{ display: 'flex', height: '100vh' }}>
  
  {/* 左サイドバー広告 (PCのみ表示) */}
  <aside className="ad-sidebar left hidden-mobile">
    <AdSenseSlot format="vertical" />
  </aside>

  {/* ゲームメインエリア */}
  <main className="game-wrapper" style={{ flex: 1, position: 'relative' }}>
    <Canvas>
      {/* Three.js Scene */}
    </Canvas>
    
    {/* UIレイヤー (HUD) */}
    <div className="ui-layer">...</div>

    {/* モバイル用アンカー広告はAdSense自動広告にお任せ、または下部に固定枠 */}
  </main>

  {/* 右サイドバー広告 (PCのみ表示) */}
  <aside className="ad-sidebar right hidden-mobile">
    <AdSenseSlot format="vertical" />
  </aside>

</div>
```

## 4. さらなる収益化：リワード広告の併用（強く推奨）

「常時表示」はPV（表示回数）で稼ぐモデルだが、単価は低い。
放置ゲームと相性が抜群なのは、**「ユーザーが自ら選んで動画を見る」リワード広告**である。

### 導入アイデア（仕様書・提案書準拠）

| 報酬内容 | タイミング | ユーザーメリット | 実装難易度 |
| :--- | :--- | :--- | :--- |
| **敗北時のリソース救済** | ゲームオーバー時 | 失うはずだったアイテムを50%→100%持ち帰れる | 低 |
| **ガチャ追加実行** | バトル中/ロビー | 装備ガチャや同僚機ガチャを追加で1回引ける | 中 |
| **倍速モード解放** | ロビー | 30分間、ゲームスピードを2倍にできる | 中 |
| **オフライン報酬2倍** | ログイン時 | 放置（閉じていた）時間の報酬を倍増 | 低 |

### 実装方針
*   AdSense H5 ゲーム広告のリワードAPI、または動画リワード特化の別Network（Unity Ads Web版など）を検討。
*   まずはAdSenseのディスプレイ広告（常時）から始め、PVが増えてきたらリワードを導入するのがスムーズ。

## 5. 注意事項・リスク対策

1.  **Googleポリシー違反（無効なトラフィック）**
    *   **NG**: 「広告をクリックして応援してね！」と誘導する（規約違反でBANされます）。
    *   **NG**: ゲームの操作ボタン（「次へ」など）の直下に広告を置き、誤タップを誘発する。
    *   **対策**: 広告とコンテンツの間には必ず十分なマージン（余白）を設ける。

2.  **パフォーマンス**
    *   広告タグの読み込みでゲームの初期ロードが遅れないよう、広告スクリプトは `defer` または動的に読み込む。
    *   Three.jsのレンダリングループに影響を与えないよう注意。

3.  **審査**
    *   作りかけのサイトや、コンテンツ（文字や画像）が少なすぎるページはAdSense審査に通らないことがある。
    *   ゲーム画面だけでなく、LP（ランディングページ）や遊び方説明ページをHTMLでしっかり作っておくことが審査通過の鍵。

---

## 6. 結論：Gocha Rumble 3D への適用

1.  **PC版**: **左右サイドバーに「スカイスクレイパー（縦長）」を常時表示**。これが最も収益性が高く、ゲーム体験を阻害しない。
2.  **スマホ版**: **ロビー画面フッターに「レクタングル（長方形）」を表示**。バトル中は没入感優先で非表示、またはアンカー広告（ユーザーが消せる）とする。
3.  **将来対応**: ユーザーが慣れてきた頃に「動画を見て報酬2倍」のリワード広告を追加する。
